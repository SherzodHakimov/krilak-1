# 2 — РЕКОМЕНДАЦИИ ПО КОДУ

> **Тип:** рекомендательное. Это подход по умолчанию для организации Angular-кода. Отклонение допустимо при обоснованной причине, но в рамках проекта важнее **единый стиль**, чем точечная «идеальность».
> **Парные документы:** `1-obyazatelnye-trebovaniya.md` (обязательное), `3-istochnik-znaniy.md` (объяснения и код).
>
> Несколько из этих практик вынесены в обязательные правила (строгая типизация, отсутствие утечек подписок, обработка ошибок, SSR-безопасность) — см. файл 1. Здесь они тоже упомянуты для полноты картины.

## Базовые принципы

**1. Separation of Concerns — разделяй ответственность.** Компонент не делает всё (UI + HTTP + маппинг + бизнес-логика + валидация). Слои:

```txt
component   → UI и события пользователя
service     → бизнес-логика
api service → HTTP-запросы
mapper      → преобразование DTO ↔ model
model/types → типы
store/facade → состояние (только если действительно нужно)
```

Компонент максимально «тонкий».

**2. Single Responsibility — одна причина для изменения.** `calculateTotalPrice(items)` только считает цену, а не форматирует валюту, сохраняет и шлёт уведомление. Имя через `and` (`loadUsersAndFilterAndShowToast()`) — сигнал дробить на `loadUsers()`, `filterUsers()`, `showSuccessToast()`.

**3. DRY, но без фанатизма.** Не дублируй одинаковую бизнес-логику, но не абстрагируй слишком рано. Два похожих куска — ещё не проблема; три — пора выносить. Небольшое повторение лучше непонятной «универсальной» функции.

**4. KISS — не усложняй раньше времени.** Для маленькой фичи не нужны `facade / store / use-case / repository / adapter / abstract class`. Часто достаточно `component / service / api service / types`. Хороший код — тот, что легко читать, менять и тестировать, а не тот, где много паттернов.

**5. YAGNI — не пиши «на будущее».** Не делай универсальный `BaseRepository<T>`, если есть один API-сервис. Абстракция — после реального повторения проблемы.

**6. Feature-based структура.** Группируй по бизнес-фичам, а не только по типам файлов:

```txt
features/
  users/    { components/  services/  models/  users.routes.ts }
  payments/ { components/  services/  models/  payments.routes.ts }
shared/ { ui/  pipes/  directives/  utils/ }
core/   { api/  auth/  interceptors/  guards/ }
```

**7. Core / Shared / Features / Layout.** Базовая структура:

```txt
core      → singleton-сервисы, auth, interceptors, guards, config
shared    → переиспользуемые UI-компоненты, pipes, directives
features  → бизнес-модули
layout    → shell, sidebar, header, main layout
```

`shared` — не мусорка: используется только в одной фиче → остаётся в фиче.

**8. Smart / Dumb components.** Smart получает данные, вызывает сервисы, управляет состоянием; Dumb только отображает через `@Input` и отдаёт события через `@Output`. `UserCardComponent` не ходит в API сам — получает готовый `user`.

**9. API отдельно от UI.** Не смешивай backend DTO и frontend model. Между ними — mapper:

```ts
type UserDto = { user_id: number; full_name: string; created_at: string };
type User    = { id: number; fullName: string; createdAt: Date };

function mapUserDtoToUser(dto: UserDto): User {
  return { id: dto.user_id, fullName: dto.full_name, createdAt: new Date(dto.created_at) };
}
```

Поменяется формат backend — меняешь mapper, а не весь UI.

**10. Ясные названия.** Имя объясняет смысл. Плохо: `getData()`, `handleClick()`, `process()`, `data`, `x`. Лучше: `loadUsers()`, `submitLoginForm()`, `calculateMonthlyPayment()`, `selectedUser`, `paymentStatus`.

**11. Минимизируй состояние.** Не храни вычислимое. Вместо параллельных `users` и `activeUsers` — выводи `activeUsers` из `users` (см. №16 про `computed()`).

**12. Предсказуемый async.** Не мешай хаотично `subscribe` / `async pipe` / промисы / вложенные подписки. Вместо вложенного `subscribe`:

```ts
this.orders$ = this.userService.getUser().pipe(
  switchMap(user => this.orderService.getOrders(user.id)),
);
```
```html
<div *ngIf="orders$ | async as orders"> ... </div>
```

**13. Бизнес-логика не в HTML.** Вместо `*ngIf="user.role === 'admin' && user.status === 'active' && ..."` — `*ngIf="canEditUser"`, а условие — в компоненте/сервисе.

**14. Consistency важнее «идеальности».** Один понятный стиль на команду лучше пяти «правильных». Договоритесь: имена файлов, расположение types, именование сервисов, где делаем mapping, как обрабатываем errors, как пишем forms, как работаем со state.

## Дополнения (современный Angular и закрытие пробелов)

**15. OnPush по умолчанию + иммутабельность.** Для dumb-компонентов: `ChangeDetectionStrategy.OnPush` перепроверяет компонент только при смене ссылки во `@Input`. Условие — не мутировать данные, а создавать новые ссылки (`[...arr]`, `{...obj}`). Самый дешёвый прирост производительности.

**16. Signals + `computed()` для производного состояния (уточнение к №11).** Геттер `get activeUsers()` пересчитывается на каждом цикле change detection, даже когда `users` не менялся. `computed()` мемоизирует:

```ts
users = signal<User[]>([]);
activeUsers = computed(() => this.users().filter(u => u.isActive));
```

**17. Управление подписками (дополнение к №12; в обязательных — файл 1).** `async pipe` по умолчанию; ручной `subscribe` — только с `takeUntilDestroyed()`:

```ts
private readonly destroyRef = inject(DestroyRef);
this.userService.getUser().pipe(takeUntilDestroyed(this.destroyRef)).subscribe(/* ... */);
```

**18. Типизированные реактивные формы (закрывает «как пишем forms» из №14).** Для нетривиальных форм — только reactive (не template-driven), через `FormBuilder` с типизацией, валидаторы в одном месте.

**19. Обработка ошибок как стратегия (закрывает «как работаем с errors» из №14; в обязательных — файл 1).** HTTP error-interceptor для сквозной обработки, типизированное состояние экрана (`loading | error | data`), ошибки не глотать, пользователю — единообразные сообщения.

**20. Interceptors для сквозной логики.** Auth-токен, обработка ошибок, индикатор загрузки — в `core/interceptors`, а не размазано по сервисам.

**21. Конфигурация через `environment`, без хардкода.** Base URL API и константы — в одном месте. Никаких магических URL и строк в коде.

**22. `@for (… track …)` / `trackBy` в списках.** Без идентификации элементов Angular перерисовывает весь список — заметно на больших таблицах.

**23. Тестируемость как следствие.** Тонкие компоненты + чистые функции для бизнес-логики (мапперы, калькуляторы) + DI = код тестируется без боли. Тяжело протестировать функцию — сигнал, что она делает слишком много (см. №2).

**24. Доступность (a11y) + семантика.** Для этого проекта совпадает с SEO: семантический HTML, `alt`, `aria`, управление фокусом улучшают и индексацию, и доступность.

**25. SSR-безопасность как принцип (в обязательных — файл 1; объяснение — файл 3, «Подводные камни»).** Каждый компонент SSR-safe: никакого прямого DOM, браузерные API только за `isPlatformBrowser()` / `afterNextRender()`.

## Топ принципов для Angular (общий)

Базовая расстановка приоритетов — применима к любому Angular-проекту:

1. Компоненты должны быть тонкими.
2. API-логика — в сервисах.
3. Бизнес-логика — не в HTML.
4. Фичи группировать по доменам.
5. DTO маппить в frontend-модели.
6. Не делать абстракции заранее.
7. Не дублировать бизнес-логику.
8. Давать понятные названия.
9. Минимизировать состояние.
10. Держать единый стиль проекта.

**Самая простая формула:** хорошая организация Angular-кода — это когда компонент отвечает за экран, сервис отвечает за логику, API-слой отвечает за запросы, а структура проекта показывает бизнес-смысл приложения.

## Топ-приоритетов для этого проекта

Та же расстановка, адаптированная под SSR/SEO-специфику (добавлены SSR-безопасность, OnPush, `computed()`):

1. Компоненты тонкие (UI ↔ логика разделены).
2. API-логика в сервисах; DTO маппится в frontend-модели.
3. Бизнес-логика не в HTML.
4. Каждый компонент SSR-safe.
5. Фичи группируются по доменам (feature-based).
6. OnPush + иммутабельность; производное состояние через `computed()`.
7. Подписки не текут (`async pipe` / `takeUntilDestroyed`).
8. Ошибки обрабатываются единообразно (interceptor + типизированное состояние).
9. Не делать абстракции заранее; не дублировать бизнес-логику.
10. Понятные названия и единый стиль во всём проекте.

Для этого проекта к формуле добавляется: всё должно работать без поломок при серверном рендеринге.
