# 1 — ОБЯЗАТЕЛЬНЫЕ ТРЕБОВАНИЯ

> **Тип:** обязательное. Нарушать нельзя. ИИ соблюдает эти правила при любой задаче по проекту.
> **Парные документы:** `2-rekomendacii-po-kodu.md` (рекомендации по стилю/архитектуре), `3-istochnik-znaniy.md` (объяснения и код).
> **Порядок для ИИ:** этот файл — всегда; рекомендации — учитывать по умолчанию; источник знаний — читать по контексту.
>
> Стек: **Angular 20+** (фронт), **NestJS** (бэкенд), **PostgreSQL** (БД). Рендеринг: **SSG (Prerender)**.
>
> Главный принцип над всеми правилами: **сложность — в решениях, простота — в реализации.**

---

## 1. Уровень и качество работы

Это governing-принцип проекта: он стоит над всеми остальными правилами и применяется к каждой задаче.

**ИИ думает и проектирует на уровне senior frontend-разработчика, но реализует на уровне middle-разработчика.** Архитектуру, выбор технических решений и предвидение проблем держит на senior-уровне; структуру проекта и код пишет простыми, неусложнёнными и легко интерпретируемыми — такими, чтобы их понял и продолжил middle-разработчик. Если разработчик на проекте сменится, новый обязан суметь самостоятельно изучить проект и начать работу **без посторонней помощи и устных пояснений**. Проект и код обязаны быть легко поддерживаемыми (supportable).

Формула: **сложность — в решениях, простота — в реализации.** Senior принимает сложное решение, но воплощает его так, чтобы оно выглядело очевидным.

### 1.1. Senior-уровень — мышление и решения

- Проектировать архитектуру и поток данных осознанно: какие слои, где границы, как фичи взаимодействуют.
- Сознательно выбирать решения и понимать их компромиссы (trade-offs), а не брать первое попавшееся.
- Заранее предвидеть граничные случаи, ошибки, поведение при SSR, производительность, безопасность.
- Правильно выбирать режим рендеринга, стратегию состояния, способ работы с API под конкретную задачу.
- Оценивать последствия решения для масштабирования и поддержки, а не только «работает сейчас».

### 1.2. Middle-уровень — структура и код

- Простой, прямой код без «умных» трюков, хитрых однострочников и неочевидной магии.
- Стандартные, узнаваемые паттерны Angular — не самодельные механизмы там, где есть штатные.
- Код читается сверху вниз и понятен с первого прочтения, без необходимости держать в голове весь проект.
- Без преждевременных и избыточных абстракций: уровень сложности соответствует размеру задачи.
- Предсказуемая структура: похожие вещи лежат в похожих местах и называются по одному шаблону.

### 1.3. Сменяемость разработчика (onboarding)

- Новый middle-разработчик может открыть проект, прочитать `README` и начать работу **без устных пояснений**.
- Любая фича находится по цепочке: URL → роут → папка → компонент.
- Единые соглашения во всём проекте (именование, расположение, подходы) — чтобы не приходилось угадывать.
- Нет «племенного знания»: всё, что нужно для работы, лежит в репозитории, а не в голове автора.

### 1.4. Поддерживаемость (supportability) — критерий приёмки

- Изменение требования затрагивает **одно** место (например, формат API меняет mapper, а не весь UI).
- Связанность между модулями минимальна и очевидна; компоненты слабо зависят друг от друга.
- Нетривиальные решения задокументированы (комментарий «почему» в коде или запись в `README`).
- Код можно расширять и чинить **без переписывания** соседних частей.
- Код пригоден к тестированию (тонкие компоненты, чистые функции, DI) — это следствие хорошей структуры, а не отдельная задача.
- Ошибки воспроизводимы и локализуемы: поведение предсказуемо, нет скрытых состояний.

## 2. Жёсткие правила кода

Это не стилистика, а правила, нарушение которых даёт баг или поломку. Стилевые/архитектурные принципы — в `2-rekomendacii-po-kodu.md`.

- **Типизация строгая:** TypeScript strict, **никаких `any`**. Все структуры и ответы API типизированы.
- **Подписки не текут:** `async pipe` по умолчанию; ручной `subscribe` — только с `takeUntilDestroyed()`; никаких подписок в конструкторе без очистки.
- **Ошибки не глотать:** у каждого экрана с запросом — состояние `loading | error | data`; ошибка обрабатывается, а не теряется.
- **Каждый компонент SSR-safe** (см. раздел 3).

---

## 3. Рендеринг и SEO

- Лендинг и статические страницы — **`RenderMode.Prerender` (SSG)**. CSR для лендинга запрещён.
- **Каждый компонент SSR-safe:** прямого обращения к DOM нет; браузерные API (`window`, `document`, `localStorage`, сторонние библиотеки с DOM) — только за `isPlatformBrowser()` или в `afterNextRender()`.
- SEO-теги выставляются **только через `SeoService`** (`Title`, `Meta`, canonical, JSON-LD). Вручную в компонентах — нет.
- `<title>` и `description` уникальны на каждой странице. Ровно один `<h1>`, корректная иерархия `h2/h3`, `alt` у изображений.
- Canonical, Open Graph, sitemap — **абсолютные URL**. OG-картинка 1200×630.
- `robots.txt` и `sitemap.xml` лежат в `public/` и доступны по корню домена.
- На сервере (этап билда/SSR) нет зависших таймеров и незавершённых подписок, блокирующих рендер.
- При серверных запросах к внутренним HTTPS-API с корпоративным/самоподписанным CA — задать `NODE_EXTRA_CA_CERTS` (путь к CA-бандлу) в окружении сборки/SSR.
- После `ng build` в консоли **ноль hydration-warnings**.

## 4. Сборка и деплой

- На хостинг загружается **результат сборки**, не исходники: содержимое `dist/<проект>/browser/`.
- Порядок: исходники → `ng build` → деплой `dist/<проект>/browser/`.
- nginx (или любая статика/CDN) — с fallback на `index.html`:

```nginx
server {
    root /var/www/landing/browser;
    location / { try_files $uri $uri/ /index.html; }
}
```

- NestJS — отдельный сервис; форма/калькулятор ходят к нему `HttpClient`-ом в браузере после гидрации.
